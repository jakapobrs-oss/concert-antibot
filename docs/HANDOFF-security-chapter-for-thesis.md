# บทที่ X การทดสอบและเสริมความปลอดภัยของระบบ (Security Review & Hardening)

## X.1 บทนำ

ระบบจองบัตรคอนเสิร์ตที่พัฒนาขึ้นเกี่ยวข้องกับ **ธุรกรรมทางการเงินจริง** (การชำระเงินผ่านพร้อมเพย์และการตรวจสลิป)
และ **การจัดสรรทรัพยากรที่มีจำกัด** (ที่นั่ง) ภายใต้สภาวะผู้ใช้จำนวนมากพร้อมกัน (flash-crowd) จึงมีความเสี่ยงด้าน
ความปลอดภัยสูงกว่าเว็บแอปพลิเคชันทั่วไป ทั้งการโกงเพื่อให้ได้บัตรฟรี การใช้บอต (bot) แย่งบัตร การยึดบัญชี
(account takeover) และการโจมตีให้ระบบล่ม (denial of service) ผู้จัดทำจึงดำเนินการตรวจสอบความปลอดภัยของโค้ด
(security review) อย่างเป็นระบบก่อนนำระบบขึ้นใช้งานจริง

## X.2 วิธีการตรวจสอบ (Methodology)

การตรวจสอบใช้แนวทาง **การตรวจข้ามผู้ให้บริการปัญญาประดิษฐ์ (cross-vendor AI review)** กล่าวคือ ใช้แบบจำลองภาษา
ขนาดใหญ่ (Large Language Model) จากคนละค่ายมาตรวจโค้ดชุดเดียวกันแล้วนำผลมาเทียบกัน ได้แก่ Claude Opus 4.8
(Anthropic) และ GPT-5.5 ผ่านเครื่องมือ Codex (OpenAI) เหตุผลที่ใช้สองตัวเพราะแบบจำลองแต่ละตัวมี "จุดบอด"
ต่างกัน การตรวจข้ามค่ายช่วยลดทั้ง **ช่องโหว่ที่มองข้าม (false negative)** และ **การรายงานผิดที่มั่นใจแต่ผิดจริง
(confident-but-wrong / false positive)**

ขั้นตอนต่อหนึ่งด้าน (แบ่งระบบเป็น 7 ด้านตามความเสี่ยง) เป็นดังนี้

1. **ตรวจด้วย AI** — ส่งไฟล์ที่เกี่ยวข้องให้ทั้งสองแบบจำลองตรวจในโหมดอ่านอย่างเดียว (read-only) พร้อมโจทย์เจาะจง
   (บทบาทผู้ตรวจความปลอดภัยระดับอาวุโส + รายการจุดที่ต้องล่า)
2. **ตรวจซ้ำกับโค้ดจริง (verify)** — ผู้จัดทำ (ร่วมกับ Claude) อ่านโค้ดจริงยืนยันทุกข้อว่าเป็นช่องโหว่จริง ไม่ใช่การเดา
   ก่อนนับเป็น finding — เป็นการป้องกัน "มั่นใจแต่ผิด" ที่สำคัญกับโค้ดเรื่องเงินเป็นพิเศษ
3. **แก้ไขที่ต้นเหตุ (fix)** — แก้ให้ตรงรากของปัญหา ไม่ใช่แค่กลบอาการ
4. **เขียนชุดทดสอบอัตโนมัติ (automated test)** — พิสูจน์ว่าการแก้ได้ผลจริง และกันการถดถอย (regression) ในอนาคต
5. **บันทึกการเปลี่ยนแปลง (commit)** แยกตามด้าน เพื่อให้ตรวจสอบย้อนกลับได้

## X.3 ผลการตรวจสอบรายด้าน

### ด้านที่ 1 — การชำระเงินและการตรวจสลิป (Payment & Slip)
พบและแก้ **6 จุด** ที่สำคัญที่สุดคือ การตรวจบัญชีผู้รับเงินเทียบ *เฉพาะเลขท้าย 4 หลัก* (จากการปกปิดเลขบัญชี
ของผู้ให้บริการตรวจสลิป) ทำให้ผู้โจมตีที่มีบัญชีเลขท้ายพ้องกับร้านสามารถโอนเข้าบัญชีตัวเองแล้วได้บัตรฟรี —
แก้โดยเพิ่มการตรวจ **ชื่อบัญชีผู้รับ** (ปลอมไม่ได้) ประกอบ. อีกจุดคือภาวะแข่งกัน (race condition) ของตัวกวาด
ออร์เดอร์หมดอายุที่อาจปล่อยที่นั่งของออร์เดอร์ใหม่ — แก้ด้วยการล็อกและอ่านข้อมูลภายในทรานแซกชันเดียวกัน

### ด้านที่ 2 — คิวและการจองที่นั่ง (Queue & Seat-hold)
พบและแก้ **8 จุด** เน้นความเป็นธรรมของคิว (fairness) และการควบคุมภาระ ตัวอย่างเด่น เช่น การปล่อยคนเข้าเลือก
ที่นั่งโดยไม่นับความจุที่ยัง active (บอตกด poll ถี่ๆ ทำให้คนทะลักเข้าพร้อมกันจนคิวไร้ผล) — แก้เป็น **admission
แบบรู้ความจุ (capacity-aware)** ที่เติมโควตาเมื่อคนข้างในเสร็จ; และการจองที่นั่งหลายตัวที่ไม่เป็นอะตอม (atomic)
ทำให้ยิงสลับลำดับแล้วล้มทั้งคู่ (griefing) — แก้เป็นสคริปต์ Lua ที่ตรวจ-แล้ว-จองทั้งชุดในคำสั่งเดียว

### ด้านที่ 3 — ระบบตรวจจับบอต (Anti-bot / Behavior / Rate-limit)
พบและแก้ **6 จุด** จุดที่ AI ตรวจข้ามค่ายช่วยได้ชัดคือ **วงจรท้าทายไม่รู้จบ (challenge loop)**: ผู้ใช้จริงที่ถูก
ระบบเข้าใจผิดว่าเป็นบอตจะถูกขอทำ CAPTCHA ซ้ำไม่หยุด (แก้ CAPTCHA ผ่านแล้วก็ยังวน) เพราะสถานะเดิมค้าง —
ทำให้ "แฟนตัวจริงเข้าไม่ได้ถาวร"; และปลายทางเขียนพฤติกรรม (behavior) ที่ไม่ต้องยืนยันตัวตน เปิดช่องเขียนทับ
ผลตัดสินของผู้อื่นและถมฐานข้อมูล — แก้โดยบังคับล็อกอิน ผูกข้อมูลกับเจ้าของจริง และตัดวงจรท้าทายเมื่อผ่าน
CAPTCHA แล้ว. ประเด็นเชิงหลักการที่บันทึกไว้ในเล่มคือ **สัญญาณฝั่งไคลเอนต์ปลอมได้ทั้งหมด — Cloudflare
Turnstile จึงเป็นด่านจริงเพียงตัวเดียว** ส่วนคะแนนพฤติกรรมเป็นเพียงชั้นเสริม (defense-in-depth)

### ด้านที่ 4 — การยืนยันตัวตนและสิทธิ์ (Authentication & Authorization)
พบและแก้ **6 จุด** มีระดับรุนแรง (HIGH) ถึง 2 จุด ได้แก่ (ก) **การยึดบัญชี (account takeover)** จากการเปิด
auto-link บัญชี Google กับบัญชีรหัสผ่านที่ยังไม่ยืนยันอีเมล — ผู้โจมตีสมัครด้วยอีเมลเหยื่อไว้ก่อน พอเหยื่อ
ล็อกอินด้วย Google ก็ถูกผูกเข้าบัญชีของผู้โจมตี; แก้โดยบังคับยืนยันอีเมลก่อนล็อกอินและปิด auto-link.
(ข) **สิทธิ์แอดมินค้างใน token**: บทบาท (role) ถูกฝังใน JSON Web Token อายุ 30 วัน โดยไม่ตรวจกับฐานข้อมูล
ทำให้แอดมินที่ถูกถอดสิทธิ์ยังเข้าถึงได้จนกว่าโทเคนจะหมดอายุ; แก้โดยตรวจบทบาทกับฐานข้อมูลจริงทุกจุดที่เป็น
แอดมิน. นอกจากนี้ยังแก้การล็อกบัญชีที่กลายเป็นเครื่องมือโจมตี (lockout DoS) และปิดช่องนับผู้ใช้ (enumeration)

### ด้านที่ 5 — ส่วนผู้ดูแลระบบและงานตามเวลา (Admin & Cron)
พบและแก้ **2 จุด** สำคัญคือ endpoint งานตามเวลา (cron) ที่กวาดออร์เดอร์ทั้งระบบ ตรวจรหัสลับ *เฉพาะเมื่อมีการ
ตั้งค่า* — หากลืมตั้งบนเครื่องจริงจะกลายเป็น endpoint เปิดโล่งให้ใครก็สั่งกวาดได้ (ผ่าน HTTP GET จึงถูกกระตุ้น
โดย crawler ได้ด้วย); แก้เป็น **fail-closed** คือถ้าไม่ตั้งรหัสลับบน production จะปฏิเสธทันที

### ด้านที่ 6 — ผู้ช่วยปัญญาประดิษฐ์ (AI Chat / Gemini)
ไม่พบช่องโหว่ระดับสูง ยืนยันว่าผู้ช่วย AI **ไม่มีสิทธิ์เข้าถึงฐานข้อมูลหรือเครื่องมือใด ๆ** และคำตอบถูกแสดงเป็น
ข้อความล้วน (ไม่มีช่อง XSS) การโจมตีแบบ prompt injection จึงทำได้เพียง "พาคุยนอกเรื่อง" ไม่นำไปสู่การรั่วข้อมูล
มีการเสริมการจำกัดขนาดประวัติแชต (กันการเผาโควตา) และให้ระบบตอบชัดเจนเมื่อยังไม่ตั้งค่า API key

### ด้านที่ 7 — โครงสร้างพื้นฐานและการตั้งค่า (Infrastructure & Config)
พบและแก้ **5 จุด** จุดเด่นที่การตรวจข้ามค่ายช่วยจับได้คือ **สวิตช์ปิดการป้องกันเงินที่ปิดได้เงียบ ๆ บน production**:
มาตรการป้องกันบัตรฟรีจากด้านที่ 1 ถูกควบคุมด้วยตัวแปรสภาพแวดล้อม (environment variable) ที่ปิดหรือไม่ตั้งค่าได้
โดยไม่มีการเตือน — เท่ากับเปิดช่องโหว่เดิมกลับมา; แก้โดยเพิ่มการเตือนดัง ๆ ตอนระบบเริ่มทำงาน (boot warning)
ทุกตัว. นอกจากนี้ยังพบว่ากุญแจลับสำหรับสุ่มลำดับคิวมีค่าเริ่มต้นที่เดาได้ (หากลืมตั้งบน production จะทำให้ลำดับคิว
คาดเดาได้ = ความเป็นธรรมเสียไป) และการเชื่อมต่อ Redis ขาดตัวจัดการข้อผิดพลาดที่อาจทำให้ระบบล่มเมื่อ Redis สะดุด

## X.4 สรุปเชิงปริมาณ

การตรวจสอบครอบคลุม 7 ด้าน ปิดช่องโหว่รวมประมาณ 35 จุด เพิ่มชุดทดสอบอัตโนมัติจาก 134 เป็น 181 ข้อ (ผ่าน
ทั้งหมด) และผ่านการทดสอบภาวะแข่งกันแบบยิงพร้อมกันจริงกับฐานข้อมูล 22 กรณี (0 ล้มเหลว) การเปลี่ยนแปลงทั้งหมด
ถูกบันทึกแยกตามด้านและตรวจสอบย้อนกลับได้

## X.5 ข้อจำกัดและงานในอนาคต

มีสองประเด็นที่บันทึกไว้แต่ยังไม่แก้โดยมีเหตุผลรองรับ ได้แก่ (1) การตั้งค่าให้ลบข้อมูลการเงินแบบลูกโซ่ (cascade
delete) ซึ่งยังไม่เป็นอันตรายเพราะระบบไม่มีเส้นทางลบผู้ใช้/คอนเสิร์ต และการแก้ต้องปรับโครงสร้างฐานข้อมูล; และ
(2) นโยบายความปลอดภัยเนื้อหา (Content-Security-Policy) ที่ยังอนุญาต inline script ตามข้อจำกัดของเฟรมเวิร์ก
ซึ่งควรยกระดับเป็นแบบ nonce ในอนาคต นอกจากนี้ก่อนขึ้นใช้งานจริงต้องตั้งค่าตัวแปรสภาพแวดล้อมด้านความปลอดภัย
ให้ครบ ซึ่งระบบมีทั้งการเตือนตอนเริ่มทำงานและสคริปต์ตรวจสอบ (`scripts/check-env.ts`) รองรับแล้ว

## X.6 บทสรุป

การนำการตรวจข้ามผู้ให้บริการ AI มาใช้พิสูจน์ว่ามีคุณค่าจริง — แบบจำลองแต่ละตัวจับช่องโหว่ที่อีกตัวมองข้าม
(เช่น วงจรท้าทายไม่รู้จบ และสวิตช์ปิดการป้องกันเงิน) ขณะเดียวกันการตรวจซ้ำกับโค้ดจริงก็ช่วยคัดข้อที่ "มั่นใจแต่ผิด"
ออกไปได้ (เช่น ตัวจำกัดอัตราที่ดูเหมือนมีช่องแต่จริง ๆ ปลอดภัย) ผลลัพธ์คือระบบที่ผ่านการเสริมความปลอดภัยครบ
ทุกชั้นตามหลัก **ป้องกันหลายชั้น (defense-in-depth)** และ **ล้มเหลวแบบปลอดภัย (fail-closed)** ก่อนนำขึ้นใช้งานจริง
